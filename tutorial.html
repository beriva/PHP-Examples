<h1 id="intermediate-php">Intermediate PHP.</h1> <h2 id="quickies-before-we-start">Quickies before we start:</h2> <p>We're going to quickly look at these.</p> <ul> <li>The Ternary Operator</li> <li>String/Variable interpolation</li> </ul> <h2 id="closures-or-anonymous-functions">Closures, or "anonymous functions".</h2> <p>Closures are functions that can be defined and passed around and used as variables. You'll see these all the time in javascript, and they're becoming a lot more widely used in PHP now, especially in frameworks.</p> <p>You define a Closure by using a function keyword like so, and assign it to a variable.</p> <pre><code>&lt;?php $myFunction = function () { echo &quot;Hello&quot;; return 200; }; </code></pre> <p>And you can use it by adding parentheses () to the variable:</p> <pre><code>&lt;?php $returnValue = $myFunction(); echo $returnValue; // 200 </code></pre> <p>They can be used as 'callbacks' - a chunk of functionality to run after or before another set of functionality.</p> <p>For example, from Laravel's router:</p> <pre><code>&lt;?php Route::get('/', function() { return 'Hello World'; }); </code></pre> <blockquote> <p>This is basically saying 'when the URL matches this, run the function passed as the second argument'</p> </blockquote> <h2 id="looking-at-objects-and-classes-again">Looking at objects and classes again.</h2> <p><img alt="Huh" src="http://i.imgur.com/dhhhvdc.jpg" /></p> <p>Objects and classes (which are PRETTY MUCH the same thing) are the foundation of modern PHP, Drupal uses them extensively under the hood, and this will become even more the case when Drupal 8 comes out. So we're just going to go over the other stuff again and introduce a few more ideas into the mix.</p> <h3 id="basic-classes">Basic Classes.</h3> <p>A class is the definition of an object - the <strong>blueprint</strong> for what the class can and can't do.</p> <pre><code>&lt;?php class Bar { public $data; public function makeMeASandwich() { echo &quot;Make your own damn sandwich!&quot;; } } </code></pre> <p>The above class has one property <code>$data</code>, and one method (function), <code>makeMeASandwich()</code>.</p> <h3 id="instantiating-creating-an-instance-of-an-object">Instantiating ("creating an instance of") an object.</h3> <p>We all know how to create an <em>instance</em> of an object by now:</p> <pre><code>$foo = new Bar(); </code></pre> <p>And we all know that we can call methods on an instance of an object, if that method exists:</p> <pre><code>$bar = $foo-&gt;makeMeASandwich(); </code></pre> <p>And that an instance of an object has <em>properties</em></p> <pre><code>$data = $foo-&gt;data; echo $data; </code></pre> <h3 id="passing-data-to-a-class">Passing data to a class.</h3> <p>When you use the <code>new</code> keyword to create a new instance of a class, you sometimes see people passing data to the class upon creating an instance, like:</p> <pre><code>$class = new FooClass($data); </code></pre> <p>When people do this, the data gets passed to the <code>__construct()</code> function inside the class, and from there, you can do with it what you want.</p> <pre><code>class FooClass { public function __construct($data) { // From here, you can do what you want with the $data. } } </code></pre> <p>Whatever arguments the <code>__construct()</code> function has, are the arguments it expects when it gets instantiated:</p> <pre><code>class FizzyPop { public function __construct($flavour, $size) { // This expects a flavour and a size. } } // Usage: $soda = new FizzyPop(&quot;Coke&quot;, &quot;2 Litres&quot;); </code></pre> <p>Arguments act just like arguments to functions, so you can supply default values to make them non-required.</p> <h3 id="using-data-inside-a-class">Using data inside a class.</h3> <p>When you're inside a class, you can use the wonderful <code>$this</code> variable to access the properties and methods of your class from inside your class functions.</p> <pre><code>class FizzyPop { public $flavour; public $size; public function __construct($flavour, $size) { // Assign the values passed to the class to the properties. $this-&gt;flavour = $flavour; $this-&gt;size = $size; } } // Usage: $soda = new FizzyPop(&quot;Orange Fanta&quot;, &quot;500ml&quot;); echo $soda-&gt;size; // Prints '500ml' echo $soda-&gt;flavour; // Prints 'Orange Fanta' </code></pre> <p>You can also call functions from inside the class:</p> <pre><code>class FizzyPop { public $flavour; public $size; public function __construct($flavour, $size) { // Assign the values passed to the class to the properties. $this-&gt;flavour = $flavour; $this-&gt;size = $size; // Call the method. $this-&gt;printFlavour(); } public function printFlavour() { echo $this-&gt;flavour; } } // Usage: $soda = new FizzyPop(&quot;Dr Pepper&quot;, &quot;500ml&quot;); // Will automatically call 'printFlavour' and print out 'Dr Pepper'; </code></pre> <h2 id="public-private-protected">Public, private, protected.</h2> <p>Methods and properties in a class can be declared as <code>public</code>, <code>private</code>, or <code>protected</code>.</p> <h3 id="public-methods-and-properties">Public methods and properties.</h3> <p>Public methods and properties can be accessed by anyone using the class, this is typically what most people default their methods and properties to, even though they probably shouldn't.</p> <pre><code>class HasPublicStuff { public $time; public myMethod() { echo 'This is public'; } } // Then, when using it: $class = new HasPublicStuff; $class-&gt;time = '10pm'; // Works. $class-&gt;myMethod(); // Works. </code></pre> <h3 id="private-methods-and-properties">Private methods and properties.</h3> <p>Private methods and properties can <strong>only be accessed from within the class they're declared in.</strong> This means that any sub-classes (we'll talk about this later on) cannot access them. Not a lot of people i know use <code>private</code> keywords in their classes.</p> <pre><code>class HasPrivateStuff { private $time; private myMethod() { echo 'This is private'; } } // Then, when using it: $class = new HasPrivateStuff; $class-&gt;time = '10pm'; // DOES NOT WORK. $class-&gt;myMethod(); // DOES NOT WORK. </code></pre> <p><img alt="Boromir" src="http://i.imgur.com/4siOckV.jpg" /></p> <h3 id="protected-methods-and-properties">Protected methods and properties.</h3> <p>Now, if you've got a property or a method that you don't want someone outside your class to use, BUT you want it to be accessible by subclasses (trust me here, in most cases, you do), then you want the protected keyword. It'll stop people outside your class using it directly, but will allow for greater flexibility down the line. Most classes you see nowadays use a mix of protected and public properties and methods.</p> <pre><code>class HasProtectedtuff { protected $time; protected myMethod() { echo 'This is protected'; } } // Then, when using it: $class = new HasProtectedStuff; $class-&gt;time = '10pm'; // DOES NOT WORK. $class-&gt;myMethod(); // DOES NOT WORK. </code></pre> <h2 id="magic-methods">Magic methods</h2> <p>MAGIC methods, that's right - you heard correctly, FREAKIN MAGIC.</p> <p>Magic methods can be placed within a class and will perform certain actions in special situations, <strong>all magic methods start with two underscores.</strong></p> <p>You'll have heard of one or two magic methods by now, at least the <code>__construct()</code> and <code>__destruct()</code> methods.</p> <p><code>__construct()</code> is called automatically when creating a new instance of an object.</p> <p><code>__destruct()</code> is called automatically when getting rid of the object, this is done by calling <code>unset()</code>, or PHP will do it when it needs memory.</p> <pre><code>class Sandwich { public function __construct() { echo 'Making a sandwich'; } public function __destruct() { echo 'Getting rid of the sandwich'; } } $s = new Sandwich; // Echoes out 'Making a sandwich'; unset($s); // Echoes out 'Getting rid of the sandwich'; </code></pre> <p>There's another couple of great magic methods: <code>__toString()</code> and <code>__invoke()</code>.</p> <h3 id="__tostring"><code>__toString()</code></h3> <p>This is used when you want to get a string representation of an object - this means that if someone passes an instance of an object to an <code>echo</code> or <code>print</code> function, or tries to use it in a string.</p> <pre><code>class FizzyPop { public $flavour; public $size; public function __construct($flavour, $size) { // Assign the values passed to the class to the properties. $this-&gt;flavour = $flavour; $this-&gt;size = $size; // Call the method. $this-&gt;printFlavour(); } public function __toString() { // Turn the flavour and size into a string and return it. return &quot;The flavour is {$this-&gt;flavour} and the size is {$this-&gt;size}.&quot;; } } // Usage: $soda = new FizzyPop(&quot;Dr Pepper&quot;, &quot;500ml&quot;); echo $soda; // Normally this would just say 'object', but now it says 'The flavour is Dr Pepper and the size is 500ml'. </code></pre> <h3 id="__invoke"><code>__invoke()</code></h3> <p>This is used when you want to call an instance of an object like a function (did you know you can call variables and objects as functions)?</p> <pre><code>class FizzyPop { public $flavour; public $size; public function __construct($flavour, $size) { // Assign the values passed to the class to the properties. $this-&gt;flavour = $flavour; $this-&gt;size = $size; // Call the method. $this-&gt;printFlavour(); } public function __invoke() { // Turn the flavour and size into a string and return it. echo &quot;The flavour is {$this-&gt;flavour} and the size is {$this-&gt;size}.&quot;; } } // Usage: $soda = new FizzyPop(&quot;Dr Pepper&quot;, &quot;500ml&quot;); echo $soda(); // Notice the brackets?, it should echo 'The flavour is Dr Pepper and the size is 500ml'. </code></pre> <h2 id="static-functions-and-properties">Static functions and properties.</h2> <p>Static functions and properties are useful for a few things:</p> <ol> <li>Grouping methods together into a re-usable set of logic.</li> <li>Replacing "global" variables by using a static varibale to store information as part of the request.</li> <li>Acting as easy-to-read facades for normal OOP functions (I'll get into this later).</li> </ol> <h3 id="what-a-static-function-looks-like">What a static function looks like.</h3> <p>You'll see static functions being called like: <code>ClassName::functionName($arguments)</code>.</p> <blockquote> <p>Note: The double-colon (<code>::</code>) used to call static methods is called the "Scope resolution operator", it also has a really weird name when throwing errors, it's called a "T_PAAMAYIM_NEKDOTAYIM" and that's hebrew for double colon, because the guy who wrote PHP originally actually spoke hebrew.</p> </blockquote> <p>And you'll see it declared very similarly to a normal class function, except it uses the <code>static</code> keyword:</p> <pre><code>class HasStaticStuff { public static function helloWorld() { echo 'Hello World'; } } // Usage: HasStaticStuff::helloWorld(); // Prints Hello world. </code></pre> <h3 id="what-a-static-variable-looks-like">What a static variable looks like.</h3> <p>In the same way that the only difference between a normal class function and a static function is the <code>static</code> keyword, the same applies to variables:</p> <pre><code>class HasStaticStuff { public static $time; } // Usage: HasStaticStuff::$time = &quot;12pm&quot;; // Set it. echo HasStaticStuff::$time; // Prints '12pm'; </code></pre> <h3 id="the-static-keyword">The <code>static</code> keyword.</h3> <p>Inside a static class, you can't use the <code>$this</code> variable - because it <strong>isn't an instance</strong>. But you still might have another static method that you want to call from within a static method, or you might want. Essentially, when working with static methods, the <code>$this</code> variable simply becomes <code>static</code> - it's a little confusing at first, but you'll soon catch on.</p> <pre><code> class HasStaticGoodness { public static function StaticOne() { // Call the other static method in this class. static::StaticTwo(); } public static function StaticTwo() { echo 'Hello, i\'m static method two'; } } // Usage: HasStaticGoodness::StaticOne(); // Will print 'Hello, i'm static method two' </code></pre> <h3 id="mixing-static-and-non-static">Mixing static and non-static.</h3> <p>This is perfectly valid and loads of people do it. In fact, some people use what we call 'static initialisers' to simplify getting an instance of an object:</p> <pre><code>class MyAwesomeClass { public $name; public $age; // Notice there's no construct function! // Static public static function initWithNameAndAge($name, $age) { $instance = new static; // Notice instead of the class name, we're using the static keyword. $instance-&gt;name = $name; $instance-&gt;age = $age; return $instance; } // Not static. public function whatsMyAgeAgain() { print $this-&gt;age; } } </code></pre> <p>Then you can do:</p> <pre><code>// Create a new instance of the object using the static method: $instance = MyAwesomeClass::initWithNameAndAge(&quot;Dan Matthews&quot;, 24); // Now we can call object methods on $instance. echo $instance-&gt;whatsMyAgeAgain(); // Prints 24. </code></pre> <h3 id="usage-example-grouping-methods-in-a-class">Usage Example: Grouping methods in a class.</h3> <p>Below is an example of methods from Stripe's <code>Stripe_Customer</code> class, they use static methods to make things accessible and easy to read, but they're grouped together in a class so that you know where to look for functions to do with Customers.</p> <pre><code>public static function retrieve($id, $apiKey=null) { $class = get_class(); return self::_scopedRetrieve($class, $id, $apiKey); } public static function all($params=null, $apiKey=null) { $class = get_class(); return self::_scopedAll($class, $params, $apiKey); } public static function create($params=null, $apiKey=null) { $class = get_class(); return self::_scopedCreate($class, $params, $apiKey); } </code></pre> <h3 id="usage-example-passing-around-data-using-static-variables">Usage Example: passing around data using static variables.</h3> <p>One of the trickiest parts of programming is persistance - keeping data accessible between different functions without having to do slow things like write it to a database or to a file.</p> <p>I came across this problem while writing CloudCPD - i needed access to a very large chunk of data that would be incredibly slow to write to a database, i could use a caching system, but at the time all we had was Drupal's cache, which just writes to the database or files, exactly what i was trying to avoid.</p> <p>Now this approach isn't recommended for huge amounts of data - i learned that the hard way, it slows sites to a standstill sometimes, but for small values, using a static property as a little 'data store' that is accessible from wherever in your script is fine.</p> <pre><code>function runThisOneFirst() { MyClass::$myValue = &quot;Hello There&quot;; } function runThisOneSecond() { echo MyClass::$myValue; } runThisOneFirst(); // Sets the value. runThisOneSecond(); // prints &quot;Hello there&quot;; </code></pre> <p>A great idea might be to use static variables / functions to replace the <code>global $user</code> object in drupal:</p> <pre><code>// Either User::$current; // Or: User::getCurrent(); </code></pre> <h2 id="inheritance">Inheritance.</h2> <p>No, not the money that your parents leave for you when they pass on. Well, kinda like that, but not at all.</p> <p>Inheritance is when one class <strong>extends</strong> another class, meaning that you have a parent class, and a child class (or <code>subclass</code>).</p> <h3 id="but-why">But Why?</h3> <p>Extending a class allows you to create specialist classes that contain methods that the parent class might not need, without perhaps adding bulk to the original class.</p> <ul> <li>Subclasses automatically have access to all the public and protected methods and properties from their parent class.</li> <li>Subclasses can override methods from the parent class.</li> <li>Used a lot in frameworks for controllers.</li> </ul> <h3 id="example">Example:</h3> <pre><code>class Foo { public $name = 'Dan Matthews'; public function whatIsMyName() { echo $this-&gt;name; } } class Bar extends Foo { } $bar = new Bar; $name = $bar-&gt;name; // $name = 'Dan Matthews'; $bar-&gt;whatIsMyName(); // Prints 'Dan Matthews'; </code></pre> <p>See how the second class has nothing in it, but i can still use the <code>whatIsMyName()</code> method? That's inheritance bitches.</p> <h3 id="a-more-extensive-example">A more extensive example.</h3> <p>Usually, websites explain inheritance using a car metaphor, and who am i to break with tradition?</p> <p>Now, say you have a class that describes a few parts of a general car, and has a few methods for general car functions.</p> <pre><code>&lt;?php class Car { protected $engine_size; protected $colour; protected $make; protected $model; public function __construct($engine_size, $colour, $make, $model) { $this-&gt;engine_size = $engine_size; $this-&gt;colour = $colour; $this-&gt;make = $make; $this-&gt;model = $model; } public function startEngine() { echo &quot;Starting engine of your {$this-&gt;colour} {$this-&gt;engine_size} {$this-&gt;make} {$this-&gt;model}&quot;&quot;; } public function honkHorn() { echo 'Honked horn…'; } } </code></pre> <blockquote> <p><strong>Note:</strong> When you see curly brackets with variables inside a string like <code>$name = "this is my {$myvariable}";</code> inside strings, this is called variable interpolation, and can only be used with double quoted strings (<code>"</code>). The curly braces won't print, but the value of the variable inside it will.</p> </blockquote> <p>Ok, so that describes pretty much every car ever, it has an engine, it can start the engine and it can honk it's horn.</p> <p>And here's how we use it:</p> <pre><code>// Create an instance of the car. $car = new Car(&quot;2 Litre&quot;, &quot;Red&quot;, &quot;VW&quot;, &quot;Golf&quot;); // Start the engine. $car-&gt;startEngine(); // This should print 'Starting engine of your red 2 litre VW Golf'. </code></pre> <h4 id="adding-new-features">Adding new features.</h4> <p>What about a more modern feature filled car? It still has all these but it has more features.</p> <ul> <li>So we <strong>could</strong> update the class to include these new features but that would add stuff to the class that we'd rarely use.</li> <li>We could also copy all of this and create a new class with the new features added.</li> <li>OR, we could create a new subclass that has the new features, but makes use of the old ones too.</li> </ul> <p>As you might have guessed, we're going to do the third to keep our code DRY.</p> <blockquote> <p><strong>Note:</strong> DRY = Don't Repeat Yourself, it's all about maximising code re-use.</p> </blockquote> <p>So here's our new <code>Modern_Car</code> class:</p> <pre><code>&lt;?php class Modern_Car extends Car { protected $sat_nav; protected $heated_seats; protected $paddle_shift; public function __construct($engine_size, $colour, $make, $model, $sat_nav, $heated_seats, $paddle_shift) { // We can still access these properties because we're extending Car. $this-&gt;engine_size = $engine_size; $this-&gt;colour = $colour; $this-&gt;make = $make; $this-&gt;model = $model; // Set our new properties. $this-&gt;sat_nav = $sat_nav; $this-&gt;heated_seats = $heated_seats; $this-&gt;paddle_shift = $paddle_shift; } public function hasHeatedSeats() { return (bool) $this-&gt;heated_seats; } public function hasSatNav() { return (bool) $this-&gt;sat_nav; } public function hasPaddleShift() { return (bool) $this-&gt;paddle_shift; } } </code></pre> <p>And here it is in use:</p> <pre><code>// New nissan qasquai with sat nav, but no heated seats and no paddleshift. $newCar = new Modern_Car(&quot;3 Litre&quot;, &quot;Black&quot;, &quot;Nissan&quot;, &quot;Qasquai&quot;, TRUE, FALSE, FALSE); if ($newCar-&gt;hasSatNav()) { // Returns TRUE; echo 'Has Sat Nav!'; } </code></pre> <blockquote> <p>Using classes like this allows us to keep maximum code re-use while still making our classes very specific for certain implementations.</p> </blockquote> <h2 id="namespaces">Namespaces</h2> <p>'Namespacing' allows you to create classes that have the same name as common tasks, but will not conflict with each other or throw "cannot redeclare function" errors.</p> <p>For example: this wouldn't be acceptable:</p> <pre><code>&lt;?php # SqliteDatabaseConnection.php Class Database { function connect() { // Connects to an sqlite database. } } ?&gt; </code></pre> <pre><code>&lt;?php # MysqlDatabaseConnection.php Class Database { function connect() { // Connects to a mysql database. } } ?&gt; </code></pre> <p>This would result in PHP telling us that two Classes have the same name, which is obviously not allowed, so instead of calling one <code>MysqlDatabase</code> or <code>SqliteDatabase</code>, we can use namespaces to logically structure our code:</p> <pre><code>&lt;?php #MysqlDatabaseConnection.php namespace Database\Connectors; class Mysql { } class Sqlite { } ?&gt; </code></pre> <p>And now we can use this class using this syntax:</p> <pre><code>// Get a Mysql Connection $database = new Database\Connectors\Mysql(); // Get a Sqlite Connection $sqliteDatabase = new Database\Connectors\Mysql(); </code></pre> <h2 id="model-view-controller">Model View Controller.</h2> <blockquote> <p>MVC Stands for Model-View-Controller, and describes the three big parts of your application.</p> </blockquote> <p>MVC is an 'architecture' that has grown massively in popularity lately, examples of things using it are:</p> <ul> <li>Symfony (PHP Framework)</li> <li>Ruby On Rails (Ruby Framework)</li> <li>Django (Python web framework)</li> <li>.NET MVC (C++ / C#)</li> <li>Laravel / FuelPHP / CakePHP / Zend Framework.</li> <li>Cocoa and Cocoa Touch (Objective-C : what iPhone and mac apps are written in)</li> </ul> <h3 id="controller">Controller</h3> <p>The controller is the heart of your application, it handles all incoming data from the HTTP Request, GET &amp; POST parameters, talks to the <strong>Model</strong> and sets up the <strong>View</strong>.</p> <h3 id="model">Model</h3> <p>This is your application's data source - it does the work of talking to the database or an API and getting the information you need to send back to the <strong>controller</strong>.</p> <h3 id="view">View</h3> <p>This is the HTML portion of your application - a view receives information from the <strong>Controller</strong> and turns it into HTML and CSS to display to the user.</p> <h3 id="router">Router</h3> <p>Although not part of the MVC acronym, all MVC web frameworks would almost be pointless without a router. A router is a piece of software that listens for HTTP connections and points them to the correct controller.</p> <h3 id="why-this-architecture-is-good">Why this architecture is good.</h3> <p>It's important because it promotes <strong>seperation of concerns</strong>, where you seperate the code that talks to your database, from the code that handles incoming requests, from the code that outputs the data to the screen.</p> <p>And why is seperation important? because imagine if you come to update the site and need to switch out large chunks of it - say you need to change the database that it uses, or you need to change the templating language it uses because of a security issue. Seperation of concerns allows you to replace just the component you need to, without</p>
